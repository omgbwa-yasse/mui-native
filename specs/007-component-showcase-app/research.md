## Decision 1: Source-Code Delivery Mechanism

**Problem**: React Native has no Node.js `fs` module. The showcase app must display each
component's internal source file (`src/components/Name/Name.tsx`) as a string at runtime,
but cannot read files from disk on a mobile device.

**Decision**: Build-time code generation script
- A Node.js script (`apps/showcase/scripts/generate-registry.ts`) runs before `npx react-native start`
- It reads every `src/components/*/index.ts{x}` in the library, collects the raw text
- It writes `apps/showcase/src/catalogue/registry.generated.ts` — a typed map of
  `componentName → sourceCodeString`
- The Metro bundler bundles the generated file as normal JS
- npm scripts: `"generate"` runs the generator; `"start"` calls `generate` first

**Rationale**:
- No native modules required
- Source code stays current — regenerate when library changes
- Type-safe output (TypeScript enum keys + string values)

**Alternatives considered**:
- Metro `loadAssets` / `require('./file.ts?raw')` — Metro does not support `?raw` out of
  the box without a custom resolver; adds build complexity with no advantage
- Manual string copy-paste — error-prone, stale on every library change; rejected
- `react-native-fs` at runtime — reads from the device bundle path, not the source tree;
  only works for bundled assets, not `.tsx` source files; rejected

## Decision 2: Navigation Library

**Decision**: `@react-navigation/native` v7 + `@react-navigation/native-stack`

**Rationale**:
- The standard React Navigation stack is the de-facto choice for RN CLI apps
- `native-stack` uses system-native navigation transitions on both iOS and Android
- Provides back-gesture and header automatically (no custom chrome needed)
- No additional native setup beyond the standard `react-native link` (autolinking)

**Alternatives considered**:
- `react-navigation/stack` (JS-only stack) — same API but uses JS-animated transitions;
  requires more setup for gestures; rejected in favour of `native-stack`
- Expo Router — requires Expo; rejected per user requirement (RN CLI)
- Custom tab-based navigation — flat category list is sufficient; nested stack better
  matches the 3-level hierarchy (Home → Category → Detail)

## Decision 3: Syntax Highlighting Library

**Decision**: `react-native-syntax-highlighter` v2 (`react-native-syntax-highlighter`)

**Rationale**:
- Wraps highlight.js 11.x; supports TypeScript/JSX language grammar out of the box
- Pure JS — no native modules required, no autolinking step
- Ships multiple themes (e.g., `atomOneDark`, `github`, etc.)
- Active maintenance; 1.5k+ GitHub stars; well-known React Native package

**Bundle impact**: ~50 kB minified (acceptable for a demo app; not a library concern)

**Alternatives considered**:
- `@highlightr/highlightr-plugin-react-native` — newer but significantly less community
  adoption and sparse documentation; rejected for stability reasons
- Manual syntax-highlighted `<Text>` spans — would require maintaining a full TypeScript
  lexer inside the showcase app; entirely out of scope; rejected
- `react-native-code-highlight` — unmaintained (last commit >3 years); rejected

## Decision 4: Component Catalogue Registry Architecture

**Decision**: A single static TypeScript registry file with explicit category groupings

Structure:
```
apps/showcase/src/catalogue/
├── registry.generated.ts      ← auto-generated by generate-registry script
├── registry.ts                ← hand-authored: categories, examples, metadata
└── types.ts                   ← shared types: Category, ComponentEntry, ExampleConfig
```

**Rationale**:
- Registry is pure data — no runtime discovery needed
- Separating generated (source strings) from authored (examples, metadata) means the
  generator does not overwrite handwritten example code
- Single import point (`catalogue/registry.ts`) for screens; no scattered imports

**Alternatives considered**:
- Filesystem-scan at runtime via `require.context` (Webpack) — not available in Metro; rejected
- Per-component `.story.tsx`-style files discovered at build time — more complex Metro
  config required; build output harder to type; rejected

## Decision 5: Session Layout Preference

**Decision**: React Context (`LayoutPreferenceContext`) with `useState`

**Rationale**:
- Layout preference is session-scoped — the spec says "persists globally for the session"
  but does NOT require persistence across app restarts
- React Context is the idiomatic solution for cross-component session state in a display app
  of this size — no external state library (Zustand, Redux) needed
- Avoids dependency on `AsyncStorage` or `@react-native-async-storage/async-storage`
  which would require native linking

**Alternatives considered**:
- `AsyncStorage` for cross-restart persistence — over-engineered for a dev showcase;
  spec explicitly says session-level only; rejected
- Zustand global store — appropriate for larger apps; unnecessary for one boolean value; rejected
- Prop drilling from root — breaks encapsulation; makes every intermediate component aware
  of layout state; rejected

## Decision 6: Priority Component Selection (≥15 for v1 full examples)

The spec requires ≥15 priority-tier components with full examples (minimum 3 per category).
Selected based on: developer first-encounter frequency + cross-category representation.

| Category | Priority Components (3 each) |
|----------|------------------------------|
| Inputs | Button, TextField, Select |
| Data Display | Text, Avatar, Chip |
| Feedback | Alert, CircularProgress, Snackbar |
| Navigation | AppBar, Tabs, Drawer |
| Layout | Card, Stack, Divider |

All 62 remaining components receive a `ExamplesPlaceholder` entry in the registry.
