## R-002: Guest Session Token Strategy

**Decision**: Short-lived JWT (24-hour expiry) with non-refreshable guest claim

**Rationale**: Guest sessions are transient by design — guests do not own accounts and should have naturally scoped access. A JWT is stateless and requires no server-side session store for normal reads. The `loginAttemptNumber` claim in the token makes each session traceable without a DB lookup on every request.

**Token payload schema**:
```json
{
  "role": "guest",
  "invitationId": "<uuid>",
  "loginAttemptNumber": 3,
  "scope": ["view", "process", "export"],
  "iat": 1720000000,
  "exp": 1720086400
}
```

**Security constraints**:
- Signed with `HS256` (minimum); `RS256` preferred for multi-service deployments
- No refresh token — when a guest JWT expires, the guest must re-enter their code (counts as a new login attempt)
- JWT secret must be in env vars, never hardcoded
- Tokens are invalidated server-side only on code revocation (maintain a revocation check via `invitationId` on protected routes)

**Alternatives considered**:
- Server-side session (Redis): Adds infrastructure dependency for auth, overkill for short-lived guest access
- Cookie-based session: Requires React Native `CookieManager`, friction on mobile

## R-003: Real-Time Agent Progress Delivery

**Decision**: Server-Sent Events (SSE) over HTTP/1.1 persistent connection

**Rationale**: Agent progress is strictly server→client (unidirectional). SSE is a perfect fit:
- Built-in auto-reconnect at the HTTP level (`EventSource` protocol)
- Text-based (`text/event-stream`), no binary overhead
- Proxies and load balancers generally handle it well
- No extra handshake round-trip vs WebSocket
- React Native supports SSE via `EventSource` polyfill (`rn-fetch-blob` or `@native-base/eventsource`)

**Event schema** (per worker update):
```text
id: <agentSessionId>-<workerId>-<seqNum>
event: worker-progress
data: {"workerId":"<uuid>","sectionId":"<uuid>","progressPercent":42,"status":"running","partialOutput":"..."}

event: supervisor-status
data: {"status":"synthesizing","workersComplete":5,"workersTotal":6}

event: session-complete
data: {"agentSessionId":"<uuid>","reportId":"<uuid>"}
```

**Alternatives considered**:
- WebSocket: Bidirectional protocol adds unnecessary complexity for unidirectional progress stream; harder to proxy
- Long polling: High server overhead, no natural reconnect, janky latency
- Polling (fixed interval): Wastes server cycles; 5 s goal requires tight polling; poor UX for fast workers

## R-004: Invitation Code Security

**Decision**: Generate with `crypto.randomInt(1_000_000, 9_999_999)` (zero-padded to 7 digits); store `bcrypt`-hashed

**Rationale**: `crypto.randomInt` is cryptographically strong (CSPRNG). Storing only the hash prevents an attacker who retrieves the DB from using stored codes. Verification is `bcrypt.compare(submittedCode.padStart(7,'0'), storedHash)`.

**Code format**: 7 numeric digits (1000000–9999999, zero-padded) — displayed to inviter as e.g. `4 391 827` for readability.

**Uniqueness**: Check for collision before inserting (SELECT WHERE code_hash = ? — negligible collision probability; retry once on collision).

**Expiry logic**: `expiresAt = createdAt + 30 days`. Expired codes fail at lookup, not at hash check.

**Alternatives considered**:
- UUID-based code: Too long for UX (7 chars selected for easy dictation/transcription)
- SHA-256 hash: Faster than bcrypt but too fast for brute-force guessing 7-digit space (10M possibilities) — bcrypt `cost=12` adds meaningful per-attempt delay to enumeration attacks
- Argon2id: Stronger than bcrypt but adds a native dependency; bcrypt is sufficient for a 7-digit space

## R-005: Rate Limiting / Lockout Implementation

**Decision**: Redis atomic `INCR` + `EXPIRE` per canonical key `ratelimit:login:<lowerCasedEmail>`

**Mechanism**:
```
On each failed login attempt:
  1. INCR ratelimit:login:<email>
  2. If new value == 1: EXPIRE ratelimit:login:<email> 900  (15 min window)
  3. If new value >= 5: return HTTP 429 + lockoutUntil = now + TTL(key)
On successful login:
  4. DEL ratelimit:login:<email>
```

**Rationale**: Atomic Redis `INCR` is race-condition-free across multiple API nodes. `EXPIRE` is set only on the first increment to avoid window extension on repeated attempts.

**Alternatives considered**:
- In-process memory: Fails across multiple server instances without sticky sessions
- DB counter (PostgreSQL): Works but adds write load to relational DB on a hot path; sub-ms Redis preferred
- Token-bucket (Redis): More sophisticated but unnecessary for a simple 5-attempt/15-min lockout rule

## R-006: AI Orchestration Pattern

**Decision**: Abstract `AgentOrchestrator` interface; default implementation using sequential async workers with `Promise.allSettled`; optional LangChain/AutoGen adapter as second implementation

**Interface**:
```typescript
interface AgentOrchestrator {
  analyze(document: string): Promise<SectionPlan>;
  runWorker(section: Section, onProgress: (p: WorkerProgress) => void): Promise<WorkerOutput>;
  synthesize(workerOutputs: WorkerOutput[]): Promise<SynthesizedDocument>;
  humanize(original: string, synthesized: SynthesizedDocument): Promise<HumanizationReport>;
}
```

**Supervisor logic**:
1. Analyze: count words, detect headings → split into N sections (see R-007)
2. For each section: spawn `WorkerAgent`, report progress via SSE emitter
3. Wait with `Promise.allSettled` — workers are independent (no ordering dependency)
4. Synthesize: stitch sections in order, resolve overlaps
5. Humanize: diff original vs synthesized, score readability

**Worker count determination** (from FR-015):
| Word count | Workers |
|---|---|
| ≤500 | 1 |
| 501–2,000 | 2–3 |
| 2,001–5,000 | 4–5 |
| >5,000 | 6 |

**Alternatives considered**:
- LangGraph / AutoGen: Powerful but adds a heavyweight Python dependency; TypeScript-native approach preferred to keep the stack consistent
- Actor model (Akka/Erlang style): Overkill for 1–6 workers with a simple linear progress model

## R-007: Section Boundary Detection

**Decision**: Two-pass algorithm — heading-based primary, word-count chunking fallback

**Algorithm**:
```text
Pass 1 (Heading detection):
  - Split on Markdown headings (^#{1,3} ) or blank-line paragraphs
  - If N detected headings matches target worker count ± 1 → use headings as boundaries

Pass 2 (Word-count chunking, if Pass 1 insufficient headings):
  - Tokenize by whitespace
  - Chunk into wordsPerSection = totalWords / targetWorkerCount
  - Break on sentence boundary (period/question/exclamation) nearest to chunk end

Edge cases:
  - Last section absorbs remainder (max 10% overage allowed)
  - Minimum section size: 50 words (collapse into adjacent if smaller)
```

**Rationale**: Documents with clear structural headings are split semantically. Plain prose falls back to proportional chunking. Both strategies avoid cutting mid-sentence.

## R-008: Subscription Validation Integration

**Decision**: Read-only query against existing billing/subscription service; feature flag gate on `multiAgentEnabled` field; no billing logic owned by this feature

**Validation sequence**:
```text
On multi-agent session start:
  1. Read subscription record for authenticated user
  2. Check status == 'active' AND billingEnd > now()
  3. Check multiAgentEnabled == true
  4. If any check fails → HTTP 403 { code: 'SUBSCRIPTION_REQUIRED' }
     (frontend navigates to subscription upsell screen)
```

**Lapsed subscription mid-session**: In-flight agent sessions MUST NOT be killed (SC-005). Subscription check only gates START of new sessions, not running sessions.

**Alternatives considered**:
- Webhook-based invalidation: Too complex for a read-only gate on session start
- Polling subscription status mid-session: Violates SC-005 constraint

## R-009: Flesch–Kincaid Scoring

**Decision**: Implement a TypeScript Flesch–Kincaid readability scorer in `shared/readability.ts` using the standard formula

**Formula**:
$$FK_{ease} = 206.835 - 1.015 \times \frac{words}{sentences} - 84.6 \times \frac{syllables}{words}$$

**Syllable counting**: Use the standard English heuristic (count vowel groups, -es/-ed/-le exceptions). This is a best-effort approximation; exact syllabication requires a dictionary lookup.

**Score interpretation** (for humanization report display):
| Score | Label |
|---|---|
| 90–100 | Very Easy |
| 70–89 | Easy |
| 60–69 | Standard |
| 30–59 | Difficult |
| 0–29 | Very Difficult |

**Rationale**: No external readability library dependency needed; the formula is simple, well-documented, and can be unit-tested deterministically.
