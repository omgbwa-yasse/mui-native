## Entities

### `Invitation`

Represents a single issued invitation code, bound to a specific recipient email.

| Field | Type | Constraints | Notes |
|---|---|---|---|
| `id` | `uuid` | PK, not null | Generated server-side |
| `inviterId` | `uuid` | FK → User.id, not null | The user who sent the invite |
| `recipientEmail` | `varchar(255)` | not null, lowercase-normalised | Bound at creation; immutable |
| `codeHash` | `varchar(255)` | not null, unique | `bcrypt(code.padStart(7,'0'), 12)` |
| `loginCount` | `integer` | not null, default 0, range 0–10 | Incremented on each successful guest login |
| `status` | `enum` | not null, default `'active'` | See state machine below |
| `createdAt` | `timestamp` | not null, default now() | |
| `expiresAt` | `timestamp` | not null, default `createdAt + 30 days` | Soft expiry checked on login |
| `revokedAt` | `timestamp` | nullable | Set when status transitions to `revoked` |
| `convertedAt` | `timestamp` | nullable | Set when invitee creates full account |

**Indexes**: `(recipientEmail, status)`, `(inviterId, status)`, `(expiresAt)` (for batch expiry jobs)

**Status State Machine**:
```
             create
               │
           [active]
          /    │    \
    revoke   expire  loginCount=10
        │      │         │
   [revoked] [expired] [converted] (all terminal)
```

### `AgentSession`

One row per multi-agent document processing run.

| Field | Type | Constraints | Notes |
|---|---|---|---|
| `id` | `uuid` | PK | |
| `userId` | `uuid` | FK → User.id, not null | Session owner |
| `documentSnapshot` | `text` | not null | Full document text at time of submission (immutable snapshot) |

| `supervisorAgentId` | `varchar(100)` | nullable | External identifier from AI provider |
| `status` | `enum` | not null, default `'analyzing'` | See state machine below |
| `startedAt` | `timestamp` | not null | |
| `completedAt` | `timestamp` | nullable | Set on `complete` or `failed` |
| `synthesizedDocument` | `text` | nullable | Set on `complete` |

**Status State Machine**:
```
 submit
    │
[analyzing] ──► [processing] ──► [synthesizing] ──► [complete]
                    │                   │
                 [failed]            [failed]
```

## Validation Rules

| Entity | Rule |
|---|---|
| `Invitation.recipientEmail` | Must pass RFC 5321 format check; lowercase-normalised before storage |
| `Invitation.loginCount` | Read-only after creation — only updated via atomic `UPDATE … SET loginCount = loginCount + 1 WHERE id = ?` |
| `Invitation.status` | Terminal states (`revoked`, `expired`, `converted`) are irreversible via application logic |
| `GuestSession.loginAttemptNumber` | Must equal `Invitation.loginCount` at time of session creation |
| `AgentSession.workerCount` | Must match the word-count band formula (R-006); validated in service layer before insert |
| `WorkerAgent.sectionIndex` | Must be unique within `agentSessionId` |
| `HumanizationReport` | Only writable after `AgentSession.status = 'complete'`; immutable after insert |

## Word-Count-to-Agent-Count Mapping

Referenced in FR-015, R-006:

| Word count | `workerCount` |
|---|---|
| 1 – 500 | 1 |
| 501 – 2,000 | 2 (≤1,000) or 3 (>1,000) |
| 2,001 – 5,000 | 4 (≤3,500) or 5 (>3,500) |
| 5,001+ | 6 |

Implementation note: formula → `Math.min(6, Math.ceil(wordCount / 950))` approximates the above; validate with exact bands in `agentCountFromWordCount(wordCount: number): number` unit test.
